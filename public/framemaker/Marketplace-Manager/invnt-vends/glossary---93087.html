Glossary

The glossary terms in this section pertain to your management of products and addons as a channel administrator — whether they are native integrations to your marketplace, pulled from a network catalog, or referral-only — including merchandising, catalog stewardship, and vendor API integration events that you might have to resolve manually when anything disrupts, delays, or prevents provisioning.

Jump to terms that start with... [ A | B | C | D | E | F | I | L | M | N | P | Q | R | S | U | W ]

A

active discounts

Configured discounts, during the time while they are valid and can be redeemed.

addon

Depending on your marketplace configuration, either our v1 or v2 implementation of supplemental integrated services.

    • ( addons v1 ) An optional supplement to a given product. Addons are ordered separately and billed separately from the products that they enhance. However:
      • Some marketplaces do not support the sale or use of addons.
      • Even on marketplaces where addons are sold, not all vendors sell them.
      • Some vendors prepare addons for some of their available products but not others.
    • ( addons v2 ) See Learn about addon versions v1 and v2

ADDON_BIND event

( addons v1 ) The integration API event type that should fire when a given addon is bound to its associated product. Successful binding determines which addons are associated with which products on your marketplace. An addon cannot be sold on your marketplace until it is bound to a product that is also sold on your marketplace. Product vendors listen for this event on their Addon notification URL endpoints.

ADDON_CANCEL event

( addons v1 ) The integration API event type that should fire when the active order for a given addon is canceled, whether manually by the consumer or automatically by the billing system. Product vendors listen for this event on their Addon notification URL endpoints.

ADDON_CHANGE event

( addons v1 ) The integration API event type that should fire when a consumer changes his or her active order for an addon. For example, the consumer might double how many units (such as hours, gigabytes, messages, records, or what have you) he or she is entitled to consume through the addon. Product vendors listen for this event on their Addon notification URL endpoints.

ADDON_ORDER event

( addons v1 ) The integration API event type that should fire when a product’s authorized user has ordered an addon to supplement it. Product vendors listen for this event on their Addon notification URL endpoints.

ADDON_UNBIND event

( addons v1 ) The integration API event type that should fire when a given addon is divorced from whichever product it was formerly able to supplement. After the addon is unbound from its parent product, it is no longer available for consumers to order on your marketplace. Product vendors listen for this event on their Addon notification URL endpoints.

attributes

Marketplace provider-defined traits — in the Attributes area at Manage > Channel > Products > Groups | Product Groups — that control if and where certain products are ever listed together in your catalog’s navigation logic. Advanced options during attribute definition include flags that can set which attributes are mandatory for publication, which attributes can be set exclusively by someone with the channel administrator role, and which attributes are visible to consumers.

B

bundles

Small collections of mutually complimentary products, promoted and sold together collectively, as one unit — typically at a discount, as compared to purchasing them all separately. In aggregate, the products that are joined together in a bundle serve at least one high-level goal of their target consumer. For example, the constituent products in a bundle could all support certain duties that are common among personnel managers at companies with fewer than 100 employees.

C

catalog

See product catalog.

categories

Marketplace provider-defined labels — in the Categories area at Manage > Channel > Products > Groups | Product Groups — to represent a product’s purpose in a very general way. For example, a product for conducting employee performance reviews and a product for payroll management might both share a “Human Resources” category label in common. And then, when a consumer clicked through the Human Resources category label on a product listings page, the returned result would include both of these products.

child marketplace

A way to characterize a corporate relationship that could connect two marketplaces.

Relative to the AppDirect platform’s network plan, a given marketplace can be the ‘child’ of another that shares its same deployment environment, and with which it is joined together as parts of some larger conglomerate. The corporate administrator who operates a parent marketplace can oversee and override what a channel administrator does on any of its child marketplaces. See sibling marketplace

customer events

The types of product integration API events that are triggered when any of the following incidents occur:

    • A user is assigned to consume one of the seats that a product subscription contract allocates.
    • A user account, for which the vendor has provisioned a seat, was changed.
    • A product loses an established customer company — either through cancellation or because a contract reaches its natural completion date without renewal.

See SUBSCRIPTION_ORDER event, SUBSCRIPTION_CHANGE event, and SUBSCRIPTION_CANCEL event.

customer posts

See user content.

D

discounts

Methods by which a product vendor or a marketplace provider can reduce consumer expense while promoting one product in particular or encouraging conversions generally. At a high level, there are two discount types:

    • fixed price discounts — The redemption value of a fixed price discount is hard coded. For example, it might be $5.00 or $10.00, supposing that the default currency on your marketplace is US dollars (USD).
    • percentage discounts — The redemption value of a percentage discount varies according to the purchase price of whichever product edition you redeem it against. For example, it might be 10 percent or 60 percent.

discount codes

Unique text strings that each identify one promotional discount in particular for purposes of configuration or redemption. Depending on how the discount is configured:

    • Either your marketplace applies its corresponding code to a given purchase automatically (in a case when the discount is correctly applicable) or the consumer must enter the code manually during purchase.
    • Its redemption might be subject to any of several possible restrictions per user, per company, or per marketplace.

Domain Provider [product types]

A product type whose underlying business logic is cross-sell or up-sell of Internet domain registration service to buyers of Microsoft business products. Thus, we only ever expose the option to buy domain service under narrow and time-limited conditions, where all of the following statements are true simultaneously:

    • You, the channel administrator , have specified a registrar to serve as your marketplace’s default provider of domain service.
    • The consumer has entered the purchase flow to obtain either a paid version or a no-cost trial version of a Microsoft business product.
    • The consumer does not have a Microsoft account, meaning, he or she has not previously obtained any hosted Microsoft product through your marketplace or elsewhere.

Many things distinguish domain service from the other products and services that your marketplace sells.

    • You can choose only one registrar to power Internet domain sales through your marketplace. The registrar that you choose is then the vendor in fact for any cross-sell or up-sell domains that consumers buy during their Microsoft product purchases on your marketplace.
    • Your marketplace shows each such consumer the domain purchasing option only one time, no matter how he or she responds to it. Every later time that the consumer enters a Microsoft purchase flow, this option is removed.
    • While some marketplace providers cultivate business relationships with registrars directly, other providers depend upon AppDirect to build multi-tier relationships on their behalf, which serve the same purpose without incurring any provider expense. Direct relationships are both labor intensive and time consuming. The domain service procedures that you are able to complete as a channel administrator depend largely on whether your contractual connection to a given registrar is direct or multi-tier.
    • You cannot sell domain service as a standalone offering. We support only the option to buy a domain while also buying a Microsoft product.

Support for this product type is enabled or disabled per marketplace.

Download (Legacy) [product types]

A product that runs locally on the consumer’s client system, rather than following the standard SaaS paradigm. Such products typically require their consumer to:

    1. Obtain an installation utility from a generated URL that is valid only once.

      2. Prove their identity.

      Enter a unique registration code.

Support for this product type is enabled or disabled per marketplace.

E

edition position

The result per product edition when a product’s vendor sets a left-to-right, top-to-bottom sequence for sorting what the Editions & Pricing tab shows on the corresponding product’s marketing profile page.

The vendor sets a given edition’s UI placement on the Editions & Pricing tab by choosing an option from the Position list in the Edition Information area at either of two navigation paths:

    • When creating an edition — Developer > Products > [ Add Product ] > Editions & Pricing | [ Add New Edition ]
    • When editing an edition — Developer > Products > {product} [ Edit ] > Editions and Pricing | Editions > { edition }


A given edition’s Position value is relative. The Position list includes exactly the same number of options as the corresponding product has configured editions. For example, with a product that has five configured editions, each edition would show five options in its Position list: 1, 2, 3, 4, and 5. The system default value is 1.

    • When all editions share their Position value in common, the corresponding product’s marketing profile page sorts editions by their creation date.
    • When some (but not all) editions share their Position value in common (such as three editions that all use the Position value 2), we sort just that subset by the creation dates of its constituent editions, and we sandwich these between editions with relatively lower Position values, before, and relatively higher Position values, after.

TIP: Alternatively, any of the vendor’s associated developers can rearrange product editions through drag-and-drop operations in the list under the left navigation’s Editions and Pricing subhead. Grab a ||| handle to start dragging the corresponding edition elsewhere in the list.

edition rank

The result per product edition when a product’s vendor ranks the relative commercial value of the corresponding product’s configured editions. This ranking is then the basis for which changes to a customer’s contract — when switching it from one edition to another (supposing that your marketplace even allows this) — constitute an upgrade and which constitute a downgrade.

The vendor ranks a given edition’s relative value by choosing an option from the Edition Rank list in the Edition Information area at either of two navigation paths:

    • When creating an edition — Developer > Products > [ Add Product ] > Editions & Pricing | [ Add New Edition ]
    • When editing an edition — Developer > Products > {product} [ Edit ] > Editions and Pricing | Editions > { edition }

A given edition’s Edition Rank value is relative. The Edition Rank list includes exactly the same number of options as the corresponding product has configured editions. For example, with a product that has three configured editions, each edition would show three options in its Edition Rank list: 1, 2, and 3. The system default value is 1.

NOTE: Before any vendor can use the edition ranking feature, you must enable it on your marketplace. Check the Use edition rank checkbox on the Subscription Changes field in the Subscriptions area at Manage > Channel > Settings > Billing Settings | Billing Functionality.

NOTE: You can allow or forbid product upgrades and downgrades globally on your marketplace. Check or clear pertinent Subscription Update Options checkboxes in the Contracts area at Manage > Channel > Settings > Billing Settings | Billing Functionality. The pertinent checkbox labels in the Contracts area are these:

• Block edition upgrades for subscriptions with active contracts
• Block edition downgrades for subscriptions with active contracts

When your marketplace forbids product upgrades and downgrades, there is no reason for vendors to set edition rank values. In this case, consistency suggests that you should disable edition ranking.

event token

A 128-bit hash to identify each event uniquely on your marketplace.

F

featured product

Any product that a marketplace provider showcases in the Featured Apps carousel.

I

integration events

Developer API events of every kind, which pertain to user/seat provisioning of any integrated product or addon.

    • ( addons v1 )
      • ADDON_BIND event
      • ADDON_CANCEL event
      • ADDON_CHANGE event
      • ADDON_ORDER event
      • ADDON_UNBIND event
    • SUBSCRIPTION_CANCEL event
    • SUBSCRIPTION_CHANGE event
    • SUBSCRIPTION_NOTICE event
    • SUBSCRIPTION_ORDER event
    • USER_ASSIGNMENT event
    • USER_UNASSIGNMENT event
    • USER_UPDATED event

L

launch approval

Vendors cannot publish products to your marketplace at their sole discretion. Instead, you decide when and if to approve their request to add a product to your catalog. Without your approval, vendors cannot sell anything through your marketplace.

M

media sources

AppDirect determines — upon request from you, the channel administrator — which media outlets your registered vendors can cite in their product marketing profiles. For example, you might decide that the only permissible sources of product coverage are The New York Times and the South China Morning Post. In this example, no marketing profile citations would be allowed for product coverage by The Washington Post or China Daily.

N

native product

A product or service whose vendor has a direct contractual relationship with the marketplace that lists it. A native product is always one of these:

    • your company (the marketplace provider)’s own proprietary product or service
    • a directly integrated third-party product or service, with whose vendor your company (the provider) has a reseller agreement that predates the signing of your AppDirect contract
    • a referral-only listing, whose billing and provisioning do not depend on AppDirect in any way.

Compare with network product.

network product

A product or service that is not API-integrated directly with the marketplace that lists it, but instead is API-integrated with a different marketplace, which AppDirect also hosts.

While it is possible for a network product to be a referral-only listing, this would be unusual.

TIP: Alternatively, for a marketplace provider whose AppDirect deployment is on premises instead of hosted, a network product is API integrated with either a sibling marketplace or a parent marketplace.

A network product is always one of these:

    • a third-party product or service that became part of the global AppDirect network catalog without any engagement from your company.
    • a third-party product or service that became part of the global AppDirect network upon request by your company after the signing of your AppDirect contract.

Compare with native product.

P

parent marketplace

A way to characterize a corporate relationship that could connect two marketplaces.

Relative to the AppDirect platform’s network plan, a given marketplace can be the ‘parent’ of others that share its same deployment environment, which are also joined together with it as parts of some larger corporate entity or conglomerate. The corporate administrator who operates a parent marketplace can oversee and override what a channel administrator does on any of its child marketplace. See sibling marketplace

popular product

Any product that a marketplace showcases automatically in its Popular Apps carousel.

product catalog

Consumer-facing product listings — which support searching, filtering, and browsing — where shoppers learn about every no-cost product and every billable product that your marketplace makes available. Your catalog can include any combination of native product and network product products, including referral-only products.

Depending on how your marketplace is configured, its catalog might be available for browsing to everyone on the Internet. Or it might be available merely to the subset of Internet users who are registered users of your marketplace.

NOTE: Some marketplaces “hide” a small number of products, which are not visible to, or purchasable by, consumers but can be purchased on their behalf by the marketplace provider or a linked reseller.

product groups

The combination of, and combined catalog-filtering effects of, three product metadata types whose varieties you define.

    • categories — Pertain to one job function, specialization, or department, such as marketing, IT, or HR, which might be sufficient unto itself as a filter while browsing your product catalog. Any defined category can have subcategories that pertain to the completion of one kind of task. For example, the “HR” category could have a performance management subcategory, while “IT” could have an orchestration subcategory.
    • attributes — Pertain to one of a product’s characteristic properties, which must be further delineated to be of any plain use as a filter while browsing your product catalog. For example, the attribute options for a hypothetical “Deployment Type” attribute could be hosted and on-prem. Or, the attribute options for an hypothetical “Operating System” attribute could be OS X , Windows , Linux , Android , and iOS.
    • customer groups — Pertain broadly to anything that adds relevance while filtering your product catalog, such as service level agreement tiers or industry verticals.

product types

The product varieties that your marketplace is configured to list. Each supported type carries its own set of implications for:

    • marketing profile content
    • pricing models and options
    • service integration
    • support for single- or multi-user implementations
    • adding custom fields to the purchase flow
    • user authentication
    • having no exposed pricing on your marketplace
    • merchandising (tagging, featuring, promotions, and sort-weighting)
    • having an Under Maintenance mode for any event type
    • having a Coming Soon mode

NOTE: In addition, your marketplace might support custom product types, developed for your exclusive use. Any such customizations are beyond the scope of this chapter.

    • Domain Provider [ product type ]
    • Download (Legacy) [ product type ]
    • Stacked Editions Web App [ product type ] — See also stacked edition
    • Web App [ product type ]
    • Web Application (manual setup) [ product type ]

promotions

Strategic campaigns and incentives that highlight and drive consumer adoption of certain products. See bundles and discounts.

Q

questions

A type of user-produced content that solicits information about a product from the product vendor. This content is meant to benefit the product’s current and potential future users.

When your marketplace is configured — in the Reviews & Questions area at Manage > Channel > Settings > Settings | Marketplace Functionality — to show user questions, registered users can submit questions about a product on its product marketing profile page — at Marketplace > { product } > Questions > [ Ask a Question ] — directed to the product’s vendor and visible on the product’s marketing profile page. Otherwise, when you configure your marketplace to hide questions, the Questions tab is removed from every product’s marketing profile page.

When your marketplace is configured — in the Reviews & Questions area at Manage > Channel > Settings > Settings | Marketplace Functionality — to hide all user content until you approve it, you can triage submitted questions one-by-one or in bulk at Manage > Channel > Marketplace > Home | Reviews & Questions. Only your approval makes a question visible on your marketplace.

The developer who integrated the product into your marketplace then receives a notification message in email if the Question Asked template is enabled in the Vendors table at Manage > Channel > Settings > Custom UI | Customer Notifications. See user content.

R

ratings

A type of user-produced content that characterizes a product’s relative quality on a fixed, five-point scale. This content is meant to benefit the product’s potential future users.

Users can submit star ratings for a product on its product marketing profile page — at Marketplace > { product } > Reviews. See user content.

However, you can set a threshold — in the Reviews and Questions area at Manage > Channel > Settings > Settings | Marketplace Functionality — for the minimum number of star ratings that a product’s users must submit collectively before any of the ratings are visible on the product’s marketing profile page.

referral-only products

Products listed in your catalog, which are sold and managed outside our platform. Such listings can be a source of supplemental income through affiliate programs or outside business agreements.

NOTE: Your AppDirect environment must run at least the listing plan before referral-only integrations are available to you.

reviews

A type of user-produced content that summarizes subjective personal experience with, and opinions about, a product’s best and worst traits. This content is meant to benefit the product’s potential future users.

When your marketplace is configured — in the Reviews & Questions area at Manage > Channel > Settings > Settings | Marketplace Functionality — to limit submission of product reviews to the billable party on a product contract, he or she can review a qualifying product on its marketing profile page at Marketplace > { product } > Reviews.

When your marketplace is configured — in the Reviews & Questions area at Manage > Channel > Settings > Settings | Marketplace Functionality — to hide all user content until you approve it, you can triage submitted reviews one-by-one or in bulk at Manage > Channel > Marketplace > Home | Reviews & Questions. Only your approval makes a review visible on your marketplace.

See user content.

S

service type

A flag that developers must set when creating a new product on a marketplace where V2 addons are enabled. One service type designates a standalone product, where the other type designates a V2 addon.

sibling marketplace

A way to characterize a corporate relationship that could connect two or more marketplaces.

A given marketplace can have ‘siblings ‘when all of them share the same parent marketplace, relative to the AppDirect platform’s network plan.

Stacked Editions Web App [product types]

A product type for applications that a user might buy and use in multiple.

The only difference between a Stacked Editions Web App and a Web App [ product type ] is that registered users can buy a Stacked Editions Web App product multiple times. The intent is that this product type should be reserved for products that can be customized uniquely per instance, such as a Microsoft SharePoint deployment, and that one company might, as a result, buy multiple times.

SUBSCRIPTION_CANCEL event

The integration API event type that should fire when a product’s authorized user has requested to cancel an active subscription according to all effective dates, penalties, and grace periods that the product contract stipulates, and relative to its billing cycle interval. Product vendors listen for this event on their Subscription Cancel notification URL endpoints.

In the case of addons ( v1 ), a SUBSCRIPTION_CANCEL event fires only when the addon’s contract ends, rather than when its authorized user requests cancellation.

SUBSCRIPTION_CHANGE event

The integration API event type that should fire when a product’s authorized user has updated, upgraded, or downgraded an active subscription. Product vendors listen for this event on their Subscription Change notification URL endpoints.

SUBSCRIPTION_NOTICE event

The integration API event type that should fire when a consumer’s rightful access to the described product has changed. Product vendors listen for this event on their Subscription Status notification URL endpoints. The event’s value is always one of these:

    • CLOSED — The order that underlies this subscription has been in a SUSPENDED or FREE_TRIAL_EXPIRED state for longer than the marketplace-configurable grace period, which is typically 1 or 2 months. The vendor should delete (not suspend) the consumer’s rightful access to the described product.
    • DEACTIVATED — Due to a late payment or possible contract violation, the consumer’s marketplace account is not currently active. We recommend that the vendor suspend his or her product access but not delete the account or its data.
    • REACTIVATED — The consumer’s account should be considered active, with typical access. This status usually indicates that the consumer paid an overdue invoice or has converted from an expired free trial to a paid seat.
    • UPCOMING_INVOICE — Invoice calculation for the consumer’s account is set to start in 24 hours. The vendor should call the Billing Usage API endpoint on your marketplace and send any updated usage data for inclusion on the upcoming invoice.

SUBSCRIPTION_ORDER event

The integration API event type that should fire when a product seat must be provisioned for a new user. Product vendors listen for this event on their Subscription Order notification URL endpoints.

U

USER_ASSIGNMENT event

The integration API event type that should fire when registered users are assigned to a product. Product vendors listen for this event on their User Assignment notification URL endpoints.

USER_UNASSIGNMENT event

The integration API event type that should fire when registered users, who were once assigned to a given product, are subsequently disconnected from it. Product vendors listen for this event on their User Unassignment notification URL endpoints.

user content

User-generated content that is specific to a given product. Such content is sometimes visible on product listing pages and sometimes visible on product marketing profile pages. See questions, ratings, and reviews.

USER_UPDATED event

The integration API event type that should fire when information changes about one of the users who is assigned to a given product. Product vendors listen for this event on their [ User Updated notification URL endpoints.

W

Web App [product types]

A traditional, fully integrated SaaS product. Products of this type support a full marketing profile, all purchasing, pricing, and billing options, management of all assignments, updates, and cancellations, and SSO from each assigned user’s MyApps page. Common examples include Google Apps, New Relic, and Sendgrid. Most products use the web app type.

Web Application (manual setup) [product types]

Instead of relying upon API automation, this nonstandard SaaS product is provisioned manually via email notifications directed to individuals at the vendor-end or the provider-end who must mark each event as resolved or completed. Each product of this type supports a full marketing profile and most purchasing, pricing, and billing options. However, many limitations apply. For example, billing does not begin at purchase, but rather when the purchase event is resolved. Metered usage pricing is not possible. SSO from the assigned user’s MyApps page is not possible. Although this type supports management of all assignments, updates, and cancellations, each event is marked PENDING until it is resolved manually.
