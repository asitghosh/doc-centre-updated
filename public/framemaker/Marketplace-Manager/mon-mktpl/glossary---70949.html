Glossary

Jump to terms that start with... [ A | C | E | I | S | U ]

A

addon_bind event

An integration event sent to the vendor’s API endpoint when a given add-on is bound to its associated product. Successful binding determines which add-ons are associated with which products on your marketplace. An add-on cannot be sold on your marketplace until it is bound to a product that is also sold on your marketplace.

addon_cancel event

An integration event sent to the vendor’s API endpoint when the active order for a given add-on is canceled, whether manually by the consumer or automatically by the billing system.

addon_change event

An integration event sent to the vendor’s API endpoint when a consumer changes his or her active order for an add-on. For example, the consumer might double how many units — such as hours, gigabytes, messages, or records — he or she is entitled to consume through the add-on.

addon_order event

An integration event sent to the vendor’s API endpoint when a product’s authorized user has ordered an add-on to supplement the corresponding product.

addon_unbind event

An integration event sent to the vendor’s API endpoint when a given add-on is divorced from whichever product it was formerly able to supplement. After an add-on is no longer bound to its former parent product, the add-on is also no longer available for consumers to order on your marketplace.

admin logs

Record entries that your marketplace writes when certain actions occur, which are of particular interest to channel administrators. Specifically, records about any of the following event types.

    • Admin joined company
    • Application created
    • Application added
    • Application published
    • Application removed
    • Application add requested
    • Application add request canceled
    • Application publish requested
    • Application publish denied
    • Assignment requested
    • Company created
    • Impersonated
    • Impersonation requested
    • Payment made
    • Payment failed
    • Quote created
    • Quote updated
    • Quote deleted
    • Reseller linked
    • Reseller unlinked
    • Sales Agent updated
    • Sales Agent removed
    • Subscription canceled
    • Subscription modified
    • Subscription purchased
    • Subscription auto-renew re-enabled
    • User assigned
    • User deleted
    • User invited
    • User removed
    • User unassigned
    • User requested subscription cancellation
    • User withdrew cancellation request

app usage logs

Record entries that your marketplace writes when any of a SaaS product’s assigned users starts a new session in it.

C

customer events

Product integration API events, which are triggered when anything affects the provisioning of, or a user’s legitimate access to, any contracted service.

E

event logs

Record entries that your marketplace writes when particular system operations or user actions meet or exceed defined thresholds regarding the provisioning of integrated products and services that they sell through it.

    • The Pending Event Logs table cites all vendor transactions of the following types (without regard for their severity level) that were not processed automatically within the expected time frame, and which might be stalled permanently unless you — the channel administrator — resolve them manually.
      • Create Subscription
      • Change Subscription
      • Cancel Subscription
      • Subscription Notice
      • User Assignment
      • User Unassignment
      • User Updated
      • Addon Order
      • AddOn Change
      • AddOn Cancel
      • AddOn Bind
      • AddOn Unbind
    • The Integration Event Logs table cites all transactions of the same types as above, including or excluding any combination of:
      • pending or completed transactions
      • events associated with the tester user role
      • events logged at the INFO (green), WARN (yellow), or ERROR (red) level
      • events associated with published or unpublished products
      • events that were or were not resolved manually
      • pending events that are or are not asynchronous

event token

A 128-bit hash to identify each product integration API event uniquely on your marketplace.

I

integration events

Events that pertain to the provisioning of integrated products and services. See customer events.

S

subscription_cancel event

An integration event sent to a vendor’s API endpoint when the billable party on a contract requests to cancel it.

subscription_change event

An integration event sent to a vendor’s API endpoint when the billable party on a contract changes its terms. Either the cited edition or the cited unit allocation differs from the previous contract.

subscription_create event

An integration event sent to a vendor’s API endpoint when:

    • Capacity must be reserved for future provisioning of N new seats (where N is a positive integer) to use a particular product edition.
    • Records must be either:
      • created to track the vendor’s new and mutual obligations with a new consumer.
      • updated to track the vendor’s changed and mutual obligations with a known consumer.

subscription_notice event

A subscription notice event tells vendors whenever one of the following event types occurs, relative to any of their integrated products:

    • DEACTIVATED — A provisioned account was deactivated. The vendor should suspend the associated user’s access — due to late payment or possible abuse, for example — but not delete any of the user’s data.
    • REACTIVATED — A previously deactivated account should be restored and its associated account should regain all of the access that he or she had prior to the deactivation.
    • CLOSED — A subscription has been in a SUSPENDED or FREE_TRIAL_EXPIRED state for a period of time that exceeds the configured grace period on your marketplace. The vendor should revoke the associated company’s access and delete all of its data.
    • UPCOMING_INVOICE — In 24 hours, your marketplace is scheduled to compute an invoice for the cited subscription. The product’s vendor should submit any appropriate calls to your marketplace via the Billing Usage API to share usage information that affects the invoice total.

U

user assignment event

An integration event sent to the vendor’s API endpoint when anyone is assigned to one of the available seats that a subscription contract allows.

user unassignment event

An integration event sent to the vendor’s API endpoint when the billable party revokes a user’s seat assignment and the seat should be made available for reassignment.

user updated event

An integration event sent to the vendor’s API endpoint when the subscription contract terms that govern a user’s provisioned access are changed. For example, the user’s reserved quantity of consumable units might now differ.
